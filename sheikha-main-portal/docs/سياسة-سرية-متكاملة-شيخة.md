# سياسة سرية متكاملة — SHEIKHA

## الهدف

بناء منظومة قوية لا تنهار عند الطوارئ، وتحمي الأسرار والبيانات الحساسة.

## 1) تصنيف السرية

- `Critical`: `JWT_SECRET`, `PASSWORD_SALT`, مفاتيح مزودي الاتصالات
- `High`: مفاتيح webhook، مفاتيح admin owner
- `Internal`: إعدادات تشغيل غير حساسة
- `Public`: إعدادات واجهة/معلومات عامة

## 2) قواعد أساسية

- ممنوع تخزين الأسرار في Git
- ممنوع طباعة الأسرار في logs
- كل سر له مالك وتاريخ تدوير
- أقل صلاحية ممكنة لكل مفتاح

## 3) التدوير (Rotation)

- أسرار Critical: كل 30 يوم أو فور أي اشتباه
- أسرار High: كل 60 يوم
- عند التدوير:
  1. إنشاء سر جديد
  2. نشر تدريجي
  3. التحقق
  4. إبطال القديم

## 4) الاستجابة لتسريب محتمل

- إبطال المفاتيح المتأثرة فورًا
- إعادة إصدار أسرار جديدة
- مراجعة كل الطلبات غير الطبيعية
- فرض إعادة تسجيل دخول إجباري عند الحاجة

## 5) حماية Webhooks

- إلزام `x-comms-webhook-key`
- رفض أي طلب بدون مفتاح صحيح
- تسجيل محاولات الفشل مع IP والتوقيت

## 6) نسخ احتياطي آمن

- نسخ مشفر للبيانات
- اختبار استرجاع دوري
- فصل صلاحيات الوصول إلى النسخ الاحتياطية

## 7) قائمة أسرار تشغيل مطلوبة

- `JWT_SECRET`
- `PASSWORD_SALT`
- `SHEIKHA_OWNER_KEY`
- `SHEIKHA_ADMIN_JWT` (تشغيلي مؤقت)
- `SHEIKHA_COMMS_WEBHOOK_KEY`
- `INFOBIP_API_KEY`
- `INFOBIP_BASE_URL`
- `INFOBIP_EMAIL_FROM`
- `INFOBIP_SMS_SENDER`
- `INFOBIP_WHATSAPP_FROM`

## 8) معيار النجاح

- لا أسرار مكشوفة في repository
- زمن تدوير أسرار حرجة < 30 دقيقة وقت الحادث
- استعادة خدمة آمنة دون فقد بيانات حرجة
