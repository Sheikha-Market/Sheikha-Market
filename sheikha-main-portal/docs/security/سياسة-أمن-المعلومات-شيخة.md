# سياسة أمن المعلومات — منظومة شيخة

## الهدف
هذه السياسة تضبط أمن المعلومات والخصوصية والملكية الفكرية في شيخة وفق مبدأ: **لا ضرر ولا ضرار**.

## النطاق
- خوادم شيخة وخدماتها.
- مستودع الكود وملفات التهيئة.
- بيانات العملاء، السجلات، والنسخ الاحتياطية.
- قنوات التوزيع (سكريبتات التثبيت والحزم).

## تصنيف البيانات
- **سري جدًا:** مفاتيح API، أسرار JWT، مفاتيح التشفير.
- **سري:** بيانات العملاء التشغيلية، سجلات التدقيق.
- **داخلي:** وثائق التشغيل الداخلية وخطط النشر.
- **عام:** صفحات الموقع والمحتوى المنشور.

## الضوابط الإلزامية
1. **إدارة الأسرار**
   - يمنع حفظ الأسرار في الكود.
   - تحفظ الأسرار في `.env` أو مدير أسرار مع صلاحية أقل امتياز.
   - يمنع طباعة الأسرار في logs.

2. **التشفير**
   - أثناء النقل: TLS 1.2+.
   - أثناء التخزين: AES-256-GCM للبيانات الحساسة.
   - تدوير مفاتيح التشفير بشكل دوري أو عند الاشتباه.

3. **التحكم بالوصول**
   - أقل صلاحية ممكنة لكل حساب/خدمة.
   - مراجعة صلاحيات المشرفين دوريًا.
   - فصل صلاحيات الإدارة عن المستخدم العام.

4. **السجلات والتدقيق**
   - تفعيل سجل أمني موحد.
   - حجب أي قيمة حساسة (token/password/key/secret).
   - حفظ السجلات بعمر احتفاظ واضح.

5. **النسخ الاحتياطي**
   - نسخ يومية مشفرة للبيانات الأساسية.
   - اختبار استرجاع شهري.
   - حفظ نسخة خارج الخادم الأساسي.

6. **سلسلة التوريد البرمجية**
   - فحص الاعتمادات (Dependency audit).
   - فحص أسرار وتسربات في CI.
   - فحص SAST أساسي قبل الإصدار.

7. **مكافحة البرمجيات الضارة**
   - فحص الملفات المرفوعة عبر ClamAV/clamd.
   - عزل الملفات المشتبه بها (quarantine).

## حماية الملكية الفكرية
- جميع الحقوق محفوظة للمالك.
- يمنع إعادة التوزيع أو إعادة البيع بدون إذن صريح.
- أي حزمة رسمية يجب أن تحمل توثيق الملكية وملف checksums.

## الامتثال والتحديث
- تُراجع السياسة كل 90 يومًا أو بعد أي حادث أمني.
- أي خرق يُسجل ويُعالَج وفق دليل الاستجابة للحوادث.

# سياسة أمن المعلومات — منظومة شيخة

## الهدف
حماية سرية وسلامة وتوافر المعلومات، وحماية الملكية الفكرية، ومنع الضرر وفق مبدأ: لا ضرر ولا ضرار.

## نطاق السياسة
- جميع الأنظمة: `server.js`، واجهات `public/`، البيانات في `data/`، وبيئات التشغيل.
- جميع المستخدمين: الإدارة، المطورون، المشغلون، والعملاء.

## التصنيف الأمني للبيانات
- **سري للغاية:** مفاتيح API، أسرار JWT، مفاتيح التشفير، بيانات الهوية.
- **سري:** بيانات العملاء، سجلات التدقيق، تقارير الأداء.
- **داخلي:** وثائق التشغيل الداخلية.
- **عام:** المحتوى التسويقي والصفحات العامة.

## الضوابط الأساسية
- جميع الأسرار تُحفظ في `.env` أو مدير أسرار، ويُمنع وضعها في الكود.
- تشفير النقل عبر HTTPS/TLS إلزامي في الإنتاج.
- تشفير البيانات الحساسة على السكون باستخدام `AES-256-GCM`.
- مبدأ أقل صلاحية (Least Privilege) لكل مفاتيح التكاملات.
- الاحتفاظ بسجلات تدقيق للعمليات الإدارية الحساسة.
- فحص تبعيات دوري + فحص أسرار + فحص SAST ضمن CI.

## إدارة الأسرار
- تدوير أسرار التشغيل دوريًا (كل 90 يومًا كحد أقصى).
- عند الاشتباه بالتسريب: إلغاء المفتاح فورًا + إنشاء مفتاح جديد + توثيق الحادثة.
- يمنع طباعة المفاتيح أو التوكنات في logs.

## النسخ الاحتياطي والاسترجاع
- نسخة يومية مشفرة للبيانات الحساسة.
- اختبار استرجاع شهري (Recovery Drill).
- الاحتفاظ بنسخ خارجية آمنة متعددة.

## الملكية الفكرية
- جميع الحقوق محفوظة للمالك.
- يمنع النسخ أو إعادة التوزيع أو إعادة البيع دون إذن صريح.
- كل إصدار رسمي يجب أن يتضمن بنود الملكية، checksum، وتوثيق الإصدار.

## المراجعة والامتثال
- مراجعة هذه السياسة كل 6 أشهر أو بعد أي حادثة أمنية حرجة.
- أي مخالفة تعامل كحادثة أمنية وتدخل ضمن خطة الاستجابة للحوادث.
