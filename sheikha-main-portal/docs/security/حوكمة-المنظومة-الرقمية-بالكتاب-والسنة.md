# حوكمة المنظومة الرقمية بالكتاب والسنة

## الهدف

تثبيت مرجعية تشغيلية موحدة تجعل التقنية خادمة للكتاب والسنة، عبر حارس شرعي رقمي يمنع المخالفات وينبه ويوثق الأحداث.

## مبادئ الحوكمة

- لا ربا
- لا غرر
- لا غش
- لا احتكار
- لا نجش
- الشفافية في الرسائل والاستجابات
- توثيق الأحداث المرفوضة للمراجعة والتحسين

## الجذر الرقمي الذكي

يتكون الجذر الرقمي الذكي من:

1. `Unified API Gateway`
2. `Smart Service Layer`
3. `Sharia Guard Layer`
4. `Identity & Role Policy`
5. `Operational Audit & Readiness`

## البذرة الرقمية الذكية

طبقة أعلى من الجذر لتوليد التفرعات الرقمية واستدامتها:

1. `Digital Seed Blueprint`
2. `Seed Germination Flow`
3. `Sustainability KPIs`

## واجهات API الحاكمة

- `GET /api/dashboards/unified/:type?`
  - مصادقة + صلاحيات + حارس شرعي
  - استجابة موحدة: `{ success, data, message, timestamp }`
- `GET /api/digital-root/status`
  - حالة الجذر الرقمي الذكي + حالة الحوكمة الشرعية
- `GET /api/digital-root/readiness`
  - تقرير جاهزية تشغيلي (لأدوار admin/owner)
- `GET /api/digital-seed/blueprint`
  - مخطط البذرة الرقمية الذكية (الجذر + التفرعات)
- `POST /api/digital-seed/germinate`
  - إنبات البذرة وتفعيل التفرعات (admin/owner/gov)
- `GET /api/digital-seed/sustainability`
  - نموذج الاستدامة ومؤشرات الاستمرارية

## سياسة التوافق المرحلي

- المسار القديم `GET /api/dashboard` لا يزال متاحًا مؤقتًا.
- يتم إرجاع ترويسات تنبيه:
  - `X-Sheikha-Deprecated: true`
  - `X-Sheikha-Use-Endpoint: /api/dashboards/unified/:type`
- تعتمد الواجهات الجديدة على المسار الموحد، مع fallback مرحلي حتى إغلاق المسار القديم.

## التدقيق التشغيلي

- عند مخالفة الضوابط الشرعية يتم:
  1. منع الطلب (`422`)
  2. رسالة تنبيه عربية واضحة
  3. تسجيل حدث تدقيق (`sharia_guard_blocked`)

## متطلبات الامتثال المستمر

- أي ميزة جديدة تمر عبر الحارس الشرعي قبل الاعتماد.
- أي endpoint جديد يستخدم مصادقة وصلاحيات موحدة.
- أي واجهة جديدة تستهلك API الموحدة بدل تشعب المسارات.

## خطة إيقاف المسارات القديمة

| المرحلة | الإجراء | الموعد المستهدف |
|---------|---------|------------------|
| 1 | إبقاء `/api/dashboard` مع ترويسات `X-Sheikha-Deprecated` و`X-Sheikha-Use-Endpoint` | جاري |
| 2 | نقل كل استهلاك الواجهات إلى `/api/dashboards/unified/:type` | خلال 30 يوم |
| 3 | إزالة fallback المسار القديم من لوحة المستخدم | بعد التحقق من عدم الاعتماد |
| 4 | إرجاع `410 Gone` من `/api/dashboard` مع رسالة توجيه | بعد 60 يوم |
| 5 | إزالة endpoint `/api/dashboard` نهائياً | بعد 90 يوم |

**شروط الانتقال:** لا يُوقف مسار قديم إلا بعد التأكد من أن جميع الواجهات تستخدم المسار الموحد ولا يوجد استهلاك خارجي للمسار القديم.
