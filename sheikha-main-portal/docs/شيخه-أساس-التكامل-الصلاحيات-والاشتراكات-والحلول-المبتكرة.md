# شيخه أساس التكامل مع الأنظمة الأخرى — صلاحيات، اشتراكات، وحلول مبتكرة

## المبدأ: شيخه هي الأساس (Hub) وليست الطرف التابع

| المفهوم | الوصف |
|---------|--------|
| **شيخه كأساس** | الأنظمة الأخرى (SAP، Oracle، ERP، منصات AI، جوجل، أمازون) تتكامل **مع** شيخه — شيخه هي المركز (Hub) الذي يحدد القواعد، الصلاحيات، والاشتراكات. |
| **الاتجاه** | التدفق: نظام العميل ← يتصل بـ ← **شيخه** ← تمنح صلاحيات ووظائف حسب الاشتراك. وليس العكس (شيخه تابعة لنظام آخر). |
| **النتيجة** | الشركة التي تريد الربط تدفع اشتراكاً أو رسوماً لشيخه؛ تحصل على صلاحيات محددة (قراءة عروض، نشر عروض، طلبات، تقارير) حسب الباقة. |

# الجزء الأول — شيخه أساس التكامل (الهيكل)

## 1.1 لماذا شيخه هي الأساس؟

| السبب | التفصيل |
|-------|---------|
| **سوق واحد متخصص** | سوق شيخه للسكراب والمعادن — المصدر الموحد للعروض والطلبات والأسعار في القطاع؛ الأنظمة الأخرى تحتاج إلى هذه البيانات والوصول إليها. |
| **قواعد موحدة** | شيخه تضع معايير التصنيف (مراحل التوريد، أنواع السكراب، التخصصات) — أي نظام يتكامل يتبع نفس القواعد لضمان توافق البيانات. |
| **هوية وثقة** | تسجيل التاجر والمصادقة تتم عبر شيخه (أو عبر نفاذ/جوجل مع ربط بحساب شيخه) — الطرف المتكامل يثق بهوية الطرف الآخر من خلال شيخه. |
| **البيانات والأسعار** | شيخه تجمع أسعاراً مرجعية وعروضاً وطلبات — الأنظمة الأخرى **تستهلك** أو **تضيف** حسب الصلاحيات الممنوحة. |

## 1.2 من يتكامل مع شيخه؟

| نوع الطرف | أمثلة | ما يريده من التكامل |
|------------|--------|---------------------|
| **مصهر / مصنع** | مصانع حديد، نحاس، ألومنيوم | استيراد عروض سكراب، إصدار طلبات شراء، تقارير مشتريات |
| **تاجر / جامع** | تجار سكراب، جامعون | نشر عروض من نظامهم، استقبال طلبات، تقارير مبيعات |
| **ناقل / لوجستي** | شركات نقل، شاحنات | استقبال طلبات نقل، تحديث حالة الشحن، حاويات |
| **منصة ERP** | SAP، Oracle، Odoo، Dynamics | ربط وحدة المشتريات أو المبيعات مع عروض وطلبات شيخه؛ أو تقديم «موديول سكراب وحاويات» لعملائهم (مثل شركات المقاولات). |
| **شركات مقاولات (عملاء أوراكل/ساب)** | مقاولات تستخدم SAP أو Oracle | لديها سكراب، تحتاج حاويات، وتبيع السكراب — من داخل نظامها عبر التكامل مع شيخه (صلاحيات محدودة: نشر عروض، طلب حاويات، بيع). |
| **منصة AI / روبوت** | ChatGPT، Claude، مساعدات داخلية | قراءة أسعار وعروض (قراءة فقط أو محدودة) للإجابة على المستخدم |
| **جهة تحليل / استثمار** | صناديق، محللون | بيانات مجمّعة (بدون أسماء) للتقارير — حسب الصلاحيات |

# الجزء الثاني — الصلاحيات (Permissions)

## 2.1 الأدوار حسب الاشتراك أو نوع العقد

| الدور | الوصف | الصلاحيات النموذجية عبر API / التكامل |
|-------|--------|----------------------------------------|
| **زائر (Visitor)** | غير مسجّل أو لم يربط نظامه | قراءة عامة: أسعار مرجعية، عدد العروض (إحصائيات عامة). لا وصول لتفاصيل العروض أو الطلبات. |
| **تاجر أساسي (Basic Trader)** | تاجر مسجّل على المنصة — بدون تكامل API | استخدام الواجهة فقط: تسجيل، عرض بضائع، لوحة تحكم. لا مفتاح API. |
| **شريك قراءة (Read Partner)** | نظام مرتبط للقراءة فقط | قراءة: عروض السوق (حسب الفلتر)، أسعار مرجعية، سلسلة التوريد والتخصصات. لا إنشاء عروض أو طلبات عبر API. |
| **شريك قراءة وكتابة (Read-Write Partner)** | نظام مرتبط للنشر والطلبات | قراءة + كتابة: نشر عروض، تحديث/حذف عروضه، إنشاء طلبات شراء، استقبال إشعارات (إن وُجدت). حدود حسب الباقة (عدد العروض، عدد الطلبات/الشهر). |
| **شريك enterprise** | شركة كبيرة بعقد مخصّص | صلاحيات أوسع: عدد عروض/طلبات أعلى أو غير محدود، Webhooks، تقارير متقدمة، دعم مخصّص، ربما وصول لبيانات مجمّعة إقليمية. |
| **مسؤول شيخه (Admin)** | إدارة المنصة | صلاحيات كاملة: إدارة المستخدمين، الاشتراكات، عرض كل العروض والطلبات، إعداد القواعد والأسعار المرجعية. |

## 2.2 صلاحيات تفصيلية حسب الوظيفة (API)

| الوظيفة | زائر | تاجر (واجهة) | شريك قراءة | شريك قراءة+كتابة | Enterprise | مسؤول |
|---------|------|--------------|------------|-------------------|------------|--------|
| قراءة أسعار مرجعية | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ |
| قراءة عروض السوق (مجمّعة/عامة) | محدود | ✓ | ✓ | ✓ | ✓ | ✓ |
| قراءة تفاصيل عروض (بعد مصادقة) | — | ✓ | ✓ | ✓ | ✓ | ✓ |
| نشر/تحديث/حذف عروض عبر API | — | — | — | ✓ (حد) | ✓ (أوسع) | ✓ |
| إنشاء طلبات شراء عبر API | — | — | — | ✓ (حد) | ✓ (أوسع) | ✓ |
| استقبال Webhooks | — | — | — | حسب الباقة | ✓ | ✓ |
| تقارير CSV/JSON (عروضي، طلباتي) | — | ✓ لوحة | — | ✓ | ✓ متقدمة | ✓ |
| إدارة مستخدمي الاشتراك (للمؤسسة) | — | — | — | — | ✓ (إن وُجد) | ✓ |

**حد = حد شهري أو سنوي حسب الباقة (مثلاً 1000 طلب/شهر، 500 عرض نشط).**

## 2.2 ب. نوعية الاشتراك حسب الصلاحية — صلاحيات محدودة وليست كاملة

| المبدأ | الوصف |
|--------|--------|
| **صلاحيات محدودة** | كل اشتراك يمنح **مجموعة محددة** من الصلاحيات — وليس «صلاحيات كاملة». المستخدم أو النظام يحصل فقط على ما اشتراه (مثلاً: عرض أسعار فقط، أو نشر عروض سكراب فقط، أو طلب حاويات فقط). |
| **لا صلاحيات كاملة إلا للمسؤول** | صلاحيات كاملة (إدارة المنصة، كل البيانات، كل المستخدمين) للمسؤول (Admin) فقط. أي شريك أو مستخدم نهائي — صلاحياته **محدودة** حسب نوع الاشتراك. |
| **حزم صلاحيات (Permission bundles)** | الاشتراك = حزمة صلاحيات: مثلاً «حزمة بائع سكراب» = نشر عروض سكراب + تحديث عروضي + استقبال طلبات؛ أو «حزمة طالب حاويات» = طلب حاويات + عرض أسعار الحاويات فقط؛ أو «حزمة عرض فقط» = قراءة أسعار مرجعية وعروض عامة دون نشر. |

### أمثلة حزم صلاحيات محدودة

| الحزمة | الصلاحيات الممنوحة (محدودة) | ما لا يُمنح |
|--------|------------------------------|-------------|
| **عرض أسعار فقط** | قراءة أسعار مرجعية (سكراب، معادن)، إحصائيات عامة | لا نشر عروض، لا طلبات، لا حاويات، لا بيانات تفصيلية لأطراف آخرين |
| **عرض + نشر عروض سكراب** | قراءة أسعار + نشر عروض سكراب (حد X عرض/شهر) + تحديث/حذف عروضي فقط | لا طلبات شراء، لا حاويات، لا وصول لعروض الآخرين بتفاصيل كاملة |
| **طلب حاويات** | طلب حاويات (حد Y طلب/شهر)، عرض توفر الحاويات في منطقتي | لا نشر عروض سكراب، لا بيع، لا إدارة حاويات الآخرين |
| **بيع سكراب (مصدر)** | نشر عروض سكراب + استقبال طلبات شراء على عروضي + تقارير مبيعاتي فقط | لا طلبات شراء من السوق (كمشتري)، لا حاويات، لا بيانات السوق الكاملة |
| **شراء + حاويات (مقاول)** | قراءة عروض سكراب + طلب حاويات + إنشاء طلبات شراء (حد معين) + تقارير مشترياتي | لا نشر عروض كتاجر، لا إدارة متجر، لا صلاحيات إدارية |

## 2.3 تطبيق الصلاحيات تقنياً

| الآلية | الوصف |
|--------|--------|
| **مفتاح API (API Key)** | كل شريك يُمنح مفتاح API مرتبط بدوره وباقته؛ الخادم يتحقق من المفتاح ويحدد الصلاحيات والحدود. |
| **OAuth2 (لاحقاً)** | للمستخدمين الذين يسجلون الدخول عبر شيخه ثم يفوّضون تطبيقاً خارجياً — التطبيق يحصل على token بصلاحيات محدودة. |
| **حدود الاستدعاء (Rate Limits)** | حسب الباقة: مثلاً 100 طلب/دقيقة للمجاني، 1000 للمدفوع، 10000 لـ enterprise. |
| **نطاق البيانات (Scope)** | مفتاح API قد يكون مقيداً بـ: قراءة فقط، أو قراءة+كتابة لعروض التاجر نفسه فقط (لا عروض الآخرين بتفاصيل كاملة إن كانت سياسة الخصوصية تمنع ذلك). |
| **منع الهندسة العكسية** | لا يُقبل ولا يُسمح بأي هندسة عكسية للمنصة أو API أو المنطق — التكامل عبر الواجهات والتراخيص الممنوحة فقط (انظر ت-5 في تعريف شيخه والقوانين). |

# الجزء الثالث — الاشتراكات ونماذج الدفع

## 3.1 الخيار الأول: الشركة تدفع اشتراكاً لشيخه

| الباقة | الفئة | السعر (مثال — قابل للتعديل) | ما يشمله |
|--------|--------|------------------------------|----------|
| **مجاني (Free)** | تاجر أو زائر | 0 | استخدام الموقع فقط؛ لا API. أو API قراءة محدود جداً (مثلاً 100 طلب/يوم). |
| **أساسي (Basic)** | تاجر صغير / مشروع | اشتراك شهري منخفض (مثلاً X ريال/شهر) | مفتاح API قراءة؛ حد معين من الاستدعاءات (مثلاً 1000/شهر). |
| **احترافي (Professional)** | تاجر أو شركة متوسطة | اشتراك شهري أو سنوي أعلى | API قراءة+كتابة؛ عدد عروض وطلبات أعلى؛ تقارير CSV/JSON؛ إشعارات بريد. |
| **Enterprise** | مصنع، مصهر، شركة كبيرة | عقد سنوي أو شهري مخصّص | حدود أعلى أو غير محدودة؛ Webhooks؛ دعم مخصّص؛ ربما SSO أو تكامل SAP/Oracle مخصّص. |

**الفكرة:** الشركة التي تريد التكامل مع شيخه تدفع الاشتراك — شيخه تمنح الصلاحيات والحدود حسب الباقة.

## 3.2 الخيار الثاني: الدفع حسب الاستخدام (Pay-as-you-go)

| البند | الوصف |
|-------|--------|
| **المبدأ** | لا اشتراك ثابت؛ الدفع حسب عدد استدعاءات API أو عدد الطلبات المنشأة أو حجم البيانات المصدرة. |
| **مثال** | أول 10 آلاف طلب API/شهر مجاني؛ بعدها X ريال لكل 1000 طلب. أو: X ريال لكل طلب شراء مُنشأ عبر API. |
| **المناسب لـ** | شركات تستخدم التكامل بشكل متقطع أو تريد تجربة قبل الالتزام باشتراك شهري. |

## 3.3 الخيار الثالث: عمولة + اشتراك (هجين)

| البند | الوصف |
|-------|--------|
| **المبدأ** | اشتراك شهري منخفض (أو مجاني) + عمولة على الصفقات المكتملة التي تمت عبر المنصة أو عبر التكامل (API). |
| **مثال** | اشتراك احترافي 500 ريال/شهر + 0.5% من قيمة كل صفقة مكتملة عبر التكامل. |
| **المناسب لـ** | جعل الدخل مرتبطاً بقيمة التداول — شيخه تستفيد من نمو مبيعات الشريك. |

## 3.4 الخيار الرابع: الشركة لا تدفع — نموذج «مجدٍ للطرفين»

| البند | الوصف |
|-------|--------|
| **المبدأ** | إذا كان وجود التاجر أو الشريك على شيخه يزيد من قيمة السوق (عروض أكثر، سيولة أعلى)، قد تقدم شيخه تكامل مجاني أو مخفف لطرف استراتيجي. |
| **شروط مقترحة** | مثلاً: مصهر كبير يسجّل ويضيف طلبات شراء — يزيد الطلب على العروض في السوق؛ شيخه تمنحه API قراءة+كتابة بحد معقول بدون اشتراك لمدة محددة، أو بشرط الإعلان عن «متكامل مع شيخه». |
| **الاستدامة** | يُحدد مسبقاً: فترة تجريبية ثم انتقال لاشتراك أو عمولة؛ أو بقاء مجاني بحدود واضحة (مثلاً 500 طلب/شهر فقط). |

## 3.5 الخيار الخامس: العكس — شيخه تدفع للمتكامل (نادر)

| البند | الوصف |
|-------|--------|
| **المبدأ** | في حالات استثنائية: لو أن منصة كبيرة (مثلاً سوق حكومي أو منصة قطاعية) تريد إدراج عروض شيخه فيها — قد تدفع شيخه رسوماً للوصول إلى بيانات شيخه (ترخيص بيانات). هنا شيخه تبقى صاحبة البيانات والأساس، والطرف الآخر يدفع لاستخدام واجهة أو بيانات مجمّعة. |

# الجزء الثالث ب — من يدفع؟ مقدم الخدمة (أوراكل/ساب/أمازون) أم المستخدم النهائي (شركة مقاولات)

## 3.ب.1 المبدأ

الاشتراك أو الرسوم قد يدفعها **أحد طرفين** (أو كليهما حسب العقد):

| من يدفع؟ | الوصف | مثال |
|----------|--------|------|
| **مقدم الخدمة (Platform Provider)** | أوراكل، ساب، أمازون، أو أي منصة ERP/سحابة — تدفع لشيخه مقابل دمج «خدمة سوق السكراب/الحاويات/البيع» في منتجها؛ وتقديمها لعملائها كجزء من اشتراكهم أو كإضافة مدفوعة. | أوراكل تدفع لشيخه ترخيص تكامل سنوي؛ عملاء أوراكل (شركات مقاولات) يحصلون على موديول «سكراب وحاويات» من داخل أوراكل دون أن يدفعوا مباشرة لشيخه. |
| **المستخدم النهائي (End User)** | شركة مقاولات أو مصنع أو تاجر تستخدم ساب أو أوراكل أو أمازون — تدفع لشيخه اشتراكاً مباشراً وتحصل على صلاحيات محددة (عرض سكراب، طلب حاويات، بيع). | شركة مقاولات لديها سكراب وتستخدم SAP — تدفع لشيخه اشتراك «بائع سكراب + حاويات» وتنشر عروضها وتطلب حاويات من داخل SAP المتكامل مع شيخه. |

## 3.ب.2 السيناريوهات

| السيناريو | من يدفع لشيخه؟ | من يستفيد؟ | ملاحظة |
|------------|-----------------|------------|---------|
| **أ. مقدم الخدمة يدفع** | أوراكل أو ساب أو أمازون تدفع لشيخه (ترخيص تكامل، اشتراك سنوي، أو حسب عدد العملاء النشطين). | عملاء أوراكل/ساب/أمازون — يحصلون على وظيفة «سكراب وحاويات وبيع» من داخل نظامهم دون فاتورة منفصلة من شيخه (أو بفاتورة مدمجة في اشتراكهم عند مقدم الخدمة). | شيخه تتعاقد مع أوراكل/ساب؛ أوراكل/ساب تدرج شيخه في «App Center» أو كإضافة — العميل النهائي قد يدفع لأوراكل/ساب فقط. |
| **ب. المستخدم النهائي يدفع** | شركة المقاولات (أو أي عميل نهائي) تدفع لشيخه اشتراكاً مباشراً. | نفس الشركة — تحصل على صلاحيات محددة (حزمة «بائع سكراب»، أو «طلب حاويات»، أو «شراء + حاويات») عبر التكامل مع SAP/Oracle الذي تستخدمه. | الفاتورة من شيخه للمستخدم النهائي؛ التكامل مع SAP/Oracle يسمح بالاستخدام من داخل النظام لكن الاشتراك مع شيخه. |
| **ج. هجين** | مقدم الخدمة يدفع «ترخيص قاعدة» لشيخه؛ والمستخدم النهائي يدفع «اشتراك صلاحيات» إضافي إذا أراد حزمة أعلى (مثلاً عدد عروض أو طلبات أكبر). | مقدم الخدمة يقدّم الوظيفة الأساسية؛ المستخدم الذي يريد حدوداً أعلى يدفع لشيخه أو لمقدم الخدمة (حسب الاتفاق). | مرن — يلائم من يريد بداية مجانية أو محدودة ثم ترقية. |

## 3.ب.3 مثال تطبيقي: عملاء أوراكل أو ساب — شركات مقاولات لديها سكراب وتحتاج حاويات وبيع

| العنصر | التفصيل |
|--------|---------|
| **الطرف الأول: مقدم الخدمة** | أوراكل (Oracle) أو ساب (SAP) — عملاؤهم شركات (مصانع، مقاولات، خدمات). |
| **الطرف الثاني: المستخدم النهائي** | شركة مقاولات تستخدم أوراكل أو ساب لإدارة مشاريعها ومشترياتها ومخزونها. لديها **سكراب** (حديد، نحاس، معدات قديمة) ناتج عن المشاريع أو الصيانة؛ تحتاج **حاويات** لجمع ونقل السكراب؛ وتريد **بيع السكراب** في السوق. |
| **الحاجة** | من داخل نفس النظام (أوراكل أو ساب) أن تتمكن شركة المقاولات من: (١) عرض سكرابها للبيع، (٢) طلب حاويات، (٣) استقبال طلبات الشراء أو الربط بسوق السكراب. دون الخروج إلى مواقع منفصلة. |
| **دور شيخه** | شيخه تقدم **التكامل** (API + صلاحيات محدودة): سوق السكراب، إدارة الحاويات، البيع والشراء — كخدمة يمكن دمجها في أوراكل أو ساب. |
| **من يدفع؟** | **خيار 1:** أوراكل/ساب تدفع لشيخه؛ تقدم لعميلها (شركة المقاولات) «موديول سكراب وحاويات» كجزء من باقة أوراكل/ساب — شركة المقاولات لا تدفع مباشرة لشيخه. **خيار 2:** شركة المقاولات تدفع لشيخه اشتراكاً (حزمة صلاحيات: بيع سكراب + طلب حاويات) وتستخدم التكامل من داخل أوراكل/ساب. **خيار 3:** أوراكل/ساب تدفع ترخيصاً لشيخه؛ شركة المقاولات تدفع «إضافة» لأوراكل/ساب أو لشيخه إذا أرادت حدوداً أعلى. |
| **الصلاحيات (محدودة)** | شركة المقاولات لا تحصل على صلاحيات كاملة: تحصل فقط على «نشر عروض سكرابي»، «طلب حاويات»، «استقبال طلبات شراء على عروضي»، «تقارير مبيعاتي» — وليس وصولاً لبيانات السوق الكاملة أو إدارة المنصة. |

# الجزء الرابع — حلول مبتكرة (كل الخيارات)

## 4.1 شيخه كمصدر الهوية (SSO — Single Sign-On)

| الفكرة | التفصيل |
|--------|---------|
| **المبدأ** | الأنظمة المتكاملة (مثلاً بوابة مصهر داخلية) تسمح للمستخدم بتسجيل الدخول **بواسطة حساب شيخه** (أو نفاذ/جوجل المرتبط بشيخه). |
| **الفائدة** | المستخدم له identity واحد؛ شيخه تصبح «مصدر الحقيقة» للهوية في قطاع السكراب والمعادن. |
| **الاشتراك** | يمكن أن تكون خدمة مدفوعة للشركات (Enterprise) أو مدمجة في باقة احترافية. |

## 4.2 شيخه كمصدر البيانات الرسمي (Data Authority)

| الفكرة | التفصيل |
|--------|---------|
| **المبدأ** | أسعار مرجعية، تصنيفات السكراب، مراحل التوريد — مصدرها الرسمي «سوق شيخه». أي تقرير أو نظام خارجي يشير إلى «مصدر: شيخه» أو «بيانات من سوق شيخه». |
| **الفائدة** | زيادة الثقة والاعتماد على شيخه؛ إمكان بيع «ترخيص بيانات» أو «تقارير بيانات» للجهات التحليلية. |
| **الاشتراك** | باقة بيانات (Data) منفصلة: دفع مقابل تصدير بيانات مجمّعة (بدون أسماء) أو أسعار تاريخية. |

## 4.3 شهادة «متكامل مع شيخه» (Sheikha Certified Integration)

| الفكرة | التفصيل |
|--------|---------|
| **المبدأ** | شركات أو مطورون ينجزون تكاملاً (مثلاً إضافة SAP تربط مع شيخه) — بعد المراجعة تحصل على شارة «متكامل مع شيخه» أو «Sheikha Certified». |
| **الفائدة** | تسويق أسهل للشركاء؛ ثقة العملاء أعلى. شيخه قد تدرجهم في صفحة «شركاء التكامل» أو «حلول معتمدة». |
| **الاشتراك** | رسوم مراجعة أو اشتراك سنوي للظهور في القائمة المعتمدة. |

## 4.4 سوق التكاملات (Integration Marketplace)

| الفكرة | التفصيل |
|--------|---------|
| **المبدأ** | صفحة أو سوق صغير تعرض «إضافات» أو «توصيفات تكامل» جاهزة: «ربط SAP مع شيخه»، «ربط Odoo مع شيخه»، «Bot واتساب لعروض شيخه». المطور أو الشركة تضع المنتج؛ المستخدم يختار ويربط (بمفتاح API). |
| **الفائدة** | انتشار أسرع؛ مطورون خارجيون يبنون حلولاً فوق شيخه؛ شيخه تأخذ نسبة أو اشتراك من المطور. |
| **الاشتراك** | عمولة على بيع التكامل أو اشتراك للمطورين لإدراج منتجاتهم. |

## 4.5 باقات حسب «الدور» وليس فقط حسب الحجم

| الفكرة | التفصيل |
|--------|---------|
| **المبدأ** | بدل باقات عامة (أساسي/احترافي/enterprise) فقط، باقات حسب الدور: «باقة مصهر»، «باقة جامع»، «باقة ناقل» — كل باقة تحتوي الوظائف والحدود المناسبة لهذا الدور. |
| **الفائدة** | تسعير أوضح وتسويق أسهل («اشتراك مصاهر»، «اشتراك ناقلين»). |
| **الاشتراك** | سعر مختلف لكل دور؛ يمكن الجمع بين دورين (مثلاً مصهر + ناقل) بباقة مركبة. |

## 4.6 رصيد API (API Credits)

| الفكرة | التفصيل |
|--------|---------|
| **المبدأ** | بدل حد شهري ثابت، المستخدم يشتري «رصيد» استدعاءات (مثلاً 10 آلاف طلب) — يُخصم مع كل استدعاء. عند انتهاء الرصيد يشتري مجدداً أو ينتقل لاشتراك. |
| **الفائدة** | مرونة للاستخدام غير المنتظم؛ دخل مقدّم عند شراء الرصيد. |
| **الاشتراك** | بيع رزم رصيد (مثلاً 10 آلاف طلب بـ X ريال). |

## 4.7 White-Label (علامة فرعية)

| الفكرة | التفصيل |
|--------|---------|
| **المبدأ** | شركة كبيرة (مثلاً اتحاد مصانع) تريد واجهة «سوق سكراب» تحت علامتها؛ الخلفية والبيانات من شيخه، الواجهة تحمل شعار العميل. |
| **الفائدة** | انتشار شيخه دون إظهار الاسم في كل مكان؛ دخل من ترخيص White-Label. |
| **الاشتراك** | عقد سنوي مرتفع لاستخدام المنصة كـ White-Label + التزام بصلاحيات وشروط الاستخدام. |

## 4.8 Webhooks ودفع إضافي للحدث

| الفكرة | التفصيل |
|--------|---------|
| **المبدأ** | في الباقات المنخفضة: Webhooks غير متاح أو محدود (مثلاً 100 حدث/شهر). في باقة أعلى أو «إضافة Webhooks» يدفع العميل مبلغاً إضافياً لعدد أحداث أعلى. |
| **الفائدة** | تمييز واضح بين الباقات؛ إيراد إضافي من العملاء الذين يحتاجون أتمتة فورية. |
| **الاشتراك** | إضافة شهرية لـ «Webhooks 5000 حدث/شهر» أو مدمج في Enterprise. |

## 4.9 نموذج «الشريك يدفع عند الجدوى»

| الفكرة | التفصيل |
|--------|---------|
| **المبدأ** | الشريك يبدأ مجاناً أو بأسعار مخفضة؛ بعد تحقيق عتبة (مثلاً 10 صفقات مكتملة أو إيراد X من المبيعات عبر شيخه) يبدأ الاشتراك أو العمولة. |
| **الفائدة** | تقليل عائق الدخول؛ الجدوى تثبت قبل الدفع. |
| **الاشتراك** | اتفاق واضح: «أول 3 أشهر مجاني ثم اشتراك احترافي» أو «عمولة بعد أول 50 صفقة». |

## 4.10 ترخيص بيانات للبحث والأكاديميا

| الفكرة | التفصيل |
|--------|---------|
| **المبدأ** | جامعات أو مراكز أبحاث تحتاج بيانات مجمّعة (أسعار، كميات، بدون أسماء) — شيخه تقدم ترخيص بيانات بأسعار منخفضة أو مجانية لأغراض بحثية مع الإشارة إلى المصدر. |
| **الفائدة** | سمعة، دراسات تستشهد بشيخه؛ قد يؤدي لاعتماد أوسع. |
| **الاشتراك** | رسوم رمزية أو مجاني مع التزام بالإشارة إلى «بيانات من سوق شيخه». |

# الجزء الخامس — ملخص التطبيق المقترح

## 5.1 ترتيب تنفيذ مقترح

| المرحلة | الإجراء |
|---------|---------|
| **1** | تأكيد نموذج الصلاحيات (أدوار: زائر، تاجر، شريك قراءة، شريك قراءة+كتابة، enterprise، مسؤول) وربطها بمفتاح API في الخادم. |
| **2** | تحديد باقات الاشتراك: مجاني، أساسي، احترافي، enterprise — مع حدود (عدد العروض، عدد طلبات API/شهر، Webhooks نعم/لا). |
| **3** | تفعيل مفتاح API مع التحقق من الصلاحية والحد (rate limit وحصة الاستدعاءات الشهرية). |
| **4** | صفحة «الأسعار والاشتراكات» في الموقع وربطها بصفحة التكامل — مع روابط «طلب مفتاح API» و«طلب اشتراك enterprise». |
| **5** | (لاحقاً) إضافة خيارات مبتكرة: رصيد API، شهادة متكامل، سوق تكاملات، White-Label — حسب الطلب والجدوى. |

## 5.2 خلاصة الرسالة

- **شيخه أساس التكامل:** الأنظمة الأخرى تتصل بها؛ شيخه تمنح الصلاحيات وتحدد الحدود حسب الاشتراك أو العقد.
- **نوعية الاشتراك حسب الصلاحية — صلاحيات محدودة وليست كاملة:** كل اشتراك = حزمة صلاحيات محددة (عرض أسعار فقط، نشر عروض سكراب، طلب حاويات، بيع سكراب، شراء+حاويات، إلخ). لا صلاحيات كاملة إلا لمسؤول شيخه.
- **من يدفع؟** إما **مقدم الخدمة** (أوراكل، ساب، أمازون) يدفع لشيخه ويقدّم الوظيفة لعملائه، أو **المستخدم النهائي** (شركة مقاولات أو مصنع) تدفع لشيخه اشتراكاً مباشراً وتحصل على صلاحيات محددة عبر التكامل مع SAP/Oracle/أمازون، أو **هجين** (مقدم الخدمة يدفع ترخيصاً؛ المستخدم يدفع إضافات عند الحاجة).
- **مثال:** عملاء أوراكل أو ساب — شركات مقاولات لديها سكراب، تحتاج حاويات، وتبيع السكراب — من داخل نظامهم عبر تكامل شيخه؛ الدفع من مقدم الخدمة أو من المستخدم النهائي حسب العقد.
- **حلول مبتكرة** تزيد الانتشار والإيراد: SSO، بيانات رسمية، شهادة متكامل، سوق تكاملات، باقات حسب الدور، رصيد API، White-Label، Webhooks مدفوعة، نموذج «ادفع عند الجدوى»، ترخيص بيانات للبحث.
