# المخطط التنفيذي وفريق العمل لإدارة المشروع والتكاليف — وتحقيق منظومة كاملة تعمل بدون موظفين آلياً

## النطاق والهدف

- **المخطط التنفيذي** — ملخص تنفيذي للتنفيذ: معالم رئيسية، قرارات، بوابات تكلفة، ربط مع المخطط العام للتنفيذ.
- **فريق العمل لإدارة المشروع والتكاليف** — هيكل مسؤوليات إدارة المشروع والتكاليف — بأقل اعتماد على موظفين: قيادة/مالك + منظومة آلية.
- **تحقيق منظومة كاملة تعمل بدون موظفين آلياً** — أن تعمل المنظومة **كاملة آلياً** قدر الإمكان — تشغيل، مراقبة، نسخ احتياطي، تقارير، ردود، إدارة تكاليف — **بدون الحاجة إلى موظفين يوميين**.

# الجزء الثاني — فريق العمل لإدارة المشروع والتكاليف (هيكل بدون موظفين يوميين)

## 2.1 المبدأ: أقل اعتماد على موظفين — الاعتماد على الآلية

| البند | الوصف |
|--------|--------|
| **الهدف** | إدارة المشروع والتكاليف بأقل اعتماد على موظفين — بحيث المنظومة تعمل **آلياً** قدر الإمكان. |
| **فريق العمل** | **قيادة المشروع / المالك** فقط — قرارات استراتيجية، أولويات، موافقات عند الحاجة؛ لا فريق تشغيل يومي. |
| **الباقي** | تشغيل، مراقبة، نسخ احتياطي، تقارير، تنبيهات — **آلي** (سكربتات، جدولة، استضافة مُدارة، أدوات سحابية). |
| **استثناءات** | تطوير ميزات جديدة أو إصلاح معقد — يمكن تعاقد مؤقت (مستقل أو شركة) عند الحاجة؛ ليس توظيفاً يومياً. |

## 2.2 هيكل المسؤوليات (بدون موظفين يوميين)

| المسؤولية | من يؤدّيها | كيف (بدون موظفين) |
|------------|------------|---------------------|
| **قيادة المشروع / المالك** | المالك أو قائد واحد | قرارات استراتيجية، مراجعة ربع سنوية/نصف سنوية، موافقات عند عتبات تكلفة أو مخاطر. |
| **تشغيل الخادم والمنصة** | آلي | استضافة مُدارة (VPS أو PaaS) — إعادة تشغيل تلقائي، مراقبة صحة (health check). |
| **مراقبة الأداء والزوار** | آلي | مؤشرات في لوحة التحكم؛ سكربت أو خدمة تجمع البيانات وتُخزّنها؛ تنبيه عند عطل (اختياري). |
| **النسخ الاحتياطي** | آلي | جدولة (cron أو ميزة سحابية) — نسخ بيانات دورية دون تدخل. |
| **التقارير (تسويق، أداء، تكلفة)** | آلي | تقارير تُولَّد من البيانات (داشبورد، أو تقرير مجدول يُصدَّر تلقائياً). |
| **إدارة التكاليف** | آلي + مراجعة مالك | تكلفة استضافة/خدمات من الفواتير أو واجهة المزود؛ تجميع في جدول أو تقرير؛ المالك يراجع ربع سنوياً. |
| **التطوير والصيانة المعقدة** | عند الحاجة فقط | تعاقد مؤقت (مستقل/شركة) عند إضافة ميزة كبيرة أو إصلاح معقد — ليس موظفاً يومياً. |

## 2.3 إدارة التكاليف بدون موظفين

| البند | التطبيق |
|--------|---------|
| **تتبع التكلفة** | فواتير الاستضافة/النطاق/الخدمات — تُجلَب من البريد أو لوحة المزود؛ أو سكربت يقرأ API الفواتير (إن وُجد) ويُدخل في جدول بسيط. |
| **تقدير مرجعي** | وثيقة تقدير-حجم-المشروع-وتكلفة-المنظومة-الرسمي.md — مرجع للتكلفة المتوقعة قبل التشغيل وخلال المراحل وقمة المنظومة. |
| **تنبيه عند تجاوز** | (اختياري) سكربت أو قيد يدوي: إذا تجاوزت التكلفة الشهرية/الربع سنوية عتبة معينة — إشعار للمالك. |
| **مراجعة دورية** | المالك يراجع التكلفة الفعلية مقابل التقدير ربع سنوياً — قرار استمرار أو تخفيض تكلفة أو توسع. |

# الجزء الثالث — تحقيق منظومة كاملة تعمل بدون موظفين آلياً

## 3.1 ما يُؤتمَت (يعمل آلياً بدون موظفين)

| المجال | ما يعمل آلياً | الآلية (مختصر) |
|--------|----------------|------------------|
| **التشغيل** | تشغيل الخادم والموقع 24/7 | استضافة مُدارة (VPS أو PaaS) — عملية الخادم تعمل باستمرار؛ إعادة تشغيل تلقائي عند التعطل (إن وُجد). |
| **النشر (Deploy)** | تحديث الكود عند الدفع أو مجدول | CI/CD (مثل GitHub Actions) أو سكربت نشر — يبني ويُنشِر دون تدخل يومي. |
| **المراقبة (Monitoring)** | صحة الخادم، توفر الموقع | health check (نقطة نهاية مثل /api/health)؛ خدمة مراقبة خارجية (uptime) تُنبه عند انقطاع. |
| **النسخ الاحتياطي** | نسخ بيانات دورية | cron أو ميزة سحابية — نسخ مجلد data/ أو قاعدة البيانات يومياً/أسبوعياً. |
| **المؤشرات والتقارير** | زوار، عمليات، أداء | APIs موجودة (/api/marketing-indicators، /api/performance-indicators)； لوحة التحكم تعرضها؛ (اختياري) سكربت يجمع ويُصدّر تقريراً دورياً. |
| **التسجيل والعروض واللوحة** | المستخدم يسجّل، يضيف عروضاً، يدير متجره | ذات خدمة (self-service) — لا موظف للرد أو الإدخال؛ المنصة تخدم المستخدم مباشرة. |
| **الامتثال والعمليات** | سجل عمليات، أرشفة | الخادم يسجّل العمليات تلقائياً؛ أرشفة مجدولة (سكربت أو خدمة). |
| **التكلفة** | تتبع فواتير واستهلاك | فواتير من مزود الاستضافة/الخدمات؛ (اختياري) تجميع في جدول أو تقرير؛ مراجعة دورية من المالك فقط. |
| **التنبيهات** | عند عطل أو عتبة | إشعار بريد أو رسالة (Telegram/Slack) عند فشل health check أو تجاوز عتبة تكلفة — المالك يتدخل عند الحاجة فقط. |

## 3.2 ما يبقى للمالك (بدون موظفين)

| المهمة | التكرار | الوصف |
|--------|----------|--------|
| **قرارات استراتيجية** | عند الحاجة | أهداف، أولويات، توسع، تعاون مع دولة أو شريك. |
| **مراجعة دورية** | ربع سنوي أو نصف سنوي | مراجعة مخطط المشروع، تقدير التكلفة، مؤشرات الأداء، قرار استمرار أو تعديل. |
| **موافقات عند عتبات** | عند الحاجة | موافقة على تجاوز عتبة تكلفة أو إطلاق مرحلة جديدة. |
| **تعاقد مؤقت للتطوير** | عند الحاجة فقط | طلب ميزة جديدة أو إصلاح معقد من مستقل أو شركة — ليس توظيفاً. |
| **رد على تنبيه** | عند ورود تنبيه عطل أو مخاطرة | فحص وإصلاح أو تفويض إصلاح — ثم العودة إلى التشغيل الآلي. |

## 3.3 مكونات المنظومة الآلية (قائمة تحقق تقنية)

| # | المكون | الوصف | الحالة المستهدفة |
|---|--------|--------|-------------------|
| 1 | **استضافة مُدارة** | خادم أو PaaS يعيد التشغيل تلقائياً عند التعطل | عند النشر الفعلي (غير localhost) |
| 2 | **نقطة صحة (Health)** | GET /api/health أو ما شابه — لمراقبة التوفر | إضافتها إن لم تكن موجودة |
| 3 | **مراقبة توفر (Uptime)** | خدمة خارجية تفحص الموقع دورياً وتُنبه عند الانقطاع | اختياري — عند النشر |
| 4 | **نسخ احتياطي مجدول** | نسخ data/ أو قاعدة البيانات يومياً/أسبوعياً | عند استقرار البيانات |
| 5 | **تسجيل وعمليات تلقائي** | الخادم يسجّل العمليات في data/operations | مُنجَز في المنظومة الحالية |
| 6 | **مؤشرات في لوحة التحكم** | زوار، عمليات، أداء — من APIs | مُدمَج |
| 7 | **ذات خدمة (Self-service)** | تسجيل، دخول، إضافة عروض، لوحة تحكم — بدون موظف | مُنجَز |
| 8 | **تنبيه عند عطل** | إشعار عند فشل health أو عتبة | اختياري |
| 9 | **تجميع تكلفة (فواتير)** | جدول أو تقرير من فواتير الاستضافة/الخدمات | يدوي بسيط أو سكربت عند التوفر |
| 10 | **نشر آلي (CI/CD)** | دفع إلى repo → بناء ونشر تلقائي | عند اعتماد سير عمل النشر |
