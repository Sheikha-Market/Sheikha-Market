# نظام حفظ العمليات والأرشفة والشبكة السحابية

## النطاق

- **حفظ العمليات بالكامل** — كل عملية تجارية (تسجيل تاجر، إضافة/تعديل/حذف عرض، تسجيل مستخدم، دخول، حاويات، إلخ) تُسجَّل فوراً دون تأخير.
- **أرشفة كاملة** — لفترة الضرورة (مثلاً 90 يوم) تُحفظ العمليات كاملة؛ بعدها تُنقَل إلى أرشيف مضغوط مع **حفظ المهم والضروري** دائماً.
- **شبكة سحابية** — منظومة حفظ سحابية مدمجة: نسخ فورية (أو شبه فورية) إلى تخزين سحابي بسرعة وكفاءة عالية، لضمان عدم فقدان البيانات عند وصول العمليات التجارية إلى مبالغ كبيرة أو عمليات كبيرة.

# الجزء الأول — ما يُحفظ ومتى

## 1.1 أنواع العمليات المُسجَّلة

| النوع | الوصف | ضروري دائماً؟ |
|-------|--------|-----------------|
| **trader_register** | تسجيل تاجر جديد | نعم |
| **listing_add** | إضافة عرض بضاعة | نعم إذا قيمة كبيرة |
| **listing_edit** | تعديل عرض | نعم إذا قيمة كبيرة |
| **listing_delete** | حذف عرض | نعم إذا كان العرض كبيراً |
| **user_register** | تسجيل مستخدم | نعم |
| **user_login** | تسجيل دخول | لا (أرشفة دورية) |
| **container_add / edit / delete** | حاويات | نعم إذا مرتبط بعملية كبيرة |
| **notification_create** | إنشاء إشعار | لا (أرشفة دورية) |

## 1.2 سياسة الحفظ

| البند | التطبيق |
|-------|---------|
| **وقت العمليات** | كل عملية تُكتَب فوراً في سجل العمليات (بدون تأخير) — كتابة غير متزامنة إن أمكن لعدم إبطاء الاستجابة. |
| **كامل لفترة الضرورة** | آخر 90 يوماً (قابل للتعديل) — عمليات كاملة في ملف سجل يومي أو ملف واحد يُدور. |
| **المهم والضروري** | تسجيل تجار، تسجيل مستخدمين، عروض تتجاوز حد قيمة (مثلاً 100 ألف ريال) — تُنسخ دائماً إلى «أرشيف ضروري» ولا تُحذف. |
| **باقي العمليات** | بعد 90 يوماً تُنقَل إلى أرشيف مضغوط (مثلاً شهرية)؛ يمكن تقليص التفاصيل للعمليات غير الضرورية. |

# الجزء الثاني — البنية التقنية

## 2.1 المسارات والملفات

| المسار/الملف | الوصف |
|--------------|--------|
| **data/operations/** | مجلد سجل العمليات والأرشيف. |
| **data/operations/current.json** أو **operations.log** | السجل الحالي (آخر 24 ساعة أو حتى حجم معين) — يُدور يومياً. |
| **data/operations/essential/** | أرشيف العمليات الضرورية فقط — يُحفظ إلى الأبد (أو حسب السياسة). |
| **data/operations/archive/YYYY-MM/**.json أو .gz | أرشيف شهري مضغوط للعمليات الكاملة. |
| **data/operations/cloud-staging/** | طابور مخرجات جاهزة للرفع السحابي — يُستهلك بواسطة عملية الرفع. |

## 2.2 الأداء (بدون تأخير، سرعة، كفاءة عالية)

| المبدأ | التطبيق |
|--------|---------|
| **عدم التأخير** | الكتابة إلى السجل غير متزامنة (setImmediate أو كيوي في الذاكرة ثم flush دوري)؛ استجابة الـ API تُرجَع فوراً قبل انتهاء الكتابة إن لزم. |
| **السرعة** | سجل واحد append-only أو كيوي في الذاكرة يُفرَّغ كل ثانية أو كل N عملية؛ تجنب فتح/إغلاق الملف في كل طلب. |
| **كفاءة عالية** | أرشيف قديم يُضغَط (gzip)؛ العمليات الضرورية فقط تُنسخ إلى essential؛ الباقي يُدار بالأرشفة الدورية. |

## 2.3 تعريف «عملية ضرورية» و«قيمة كبيرة»

| البند | التعريف (قابل للتعديل) |
|-------|-------------------------|
| **عملية ضرورية** | trader_register، user_register؛ وأي listing_add/edit/delete تتجاوز فيها القيمة (price × quantity أو price لوحدها) حداً معيّناً (مثلاً 100,000 ريال). |
| **قيمة كبيرة** | حد اختياري في الإعدادات (مثلاً OPERATIONS_ESSENTIAL_VALUE_THRESHOLD=100000) — أي عرض أو صفقة تتجاوز هذا المبلغ تُعتبر ضرورية وتُنسخ إلى essential. |

# الجزء الثالث — الشبكة السحابية

## 3.1 الهدف

- عند وصول **عمليات تجارية كبيرة** أو **مبالغ كبيرة** — يجب أن تكون هناك **شبكة سحابية** و**منظومة حفظ** للمعلومات **بالكامل**، **بدون تأخير**، **بسرعة** و**كفاءة عالية**.

## 3.2 المكونات

| المكون | الوصف |
|--------|--------|
| **طابور رفع سحابي** | كل عملية مُسجَّلة (أو كل دفعة) تُضاف إلى طابور محلي؛ عملية خلفية ترفع إلى السحابة. |
| **وجهة السحابة** | إما (أ) API ذاتي (POST إلى CLOUD_SYNC_URL) أو (ب) تخزين متوافق مع S3 (مثل AWS S3، MinIO، أو خدمة سحابية أخرى) — عبر مفاتيح بيئة. |
| **عدم التأخير** | الرفع السحابي يحدث في الخلفية؛ عدم انتظار الرفع لإرجاع استجابة المستخدم. |
| **سرعة وكفاءة** | دفعات (batch) كل N عملية أو كل T ثانية؛ ضغط (gzip) قبل الرفع إن أمكن؛ إعادة محاولة عند الفشل. |

## 3.3 التكامل

| البند | التطبيق |
|-------|---------|
| **تفعيل السحابة** | متغير بيئة CLOUD_SYNC_ENABLED=true و CLOUD_SYNC_URL أو S3_* — عند التفعيل تُضاف السجلات إلى طابور الرفع. |
| **S3 أو متوافق** | BUCKET، REGION، ACCESS_KEY، SECRET (أو ENDPOINT لـ MinIO) — رفع ملفات من cloud-staging إلى السحابة. |
| **تكامل مع المنظومة** | نفس سجل العمليات هو المصدر؛ النسخ السحابي نسخة إضافية وليست بديلاً عن الحفظ المحلي. |

# الجزء الرابع — التفعيل والتكامل مع البوابة

## 4.1 ما يُفعَّل في الخادم (server.js)

| الوظيفة | الوصف |
|---------|--------|
| **تسجيل كل عملية** | بعد كل saveJSON (traders، listings، users، containers) — استدعاء logOperation(type, entity, payloadSummary, options). |
| **حفظ فوري** | append إلى ملف سجل أو إضافة إلى كيوي في الذاكرة + flush دوري (مثلاً كل 1 ثانية أو كل 50 عملية). |
| **تصنيف ضروري** | إذا العملية trader_register أو user_register أو listing بقيمة > حد — نسخ إلى essential. |
| **أرشفة دورية** | مهمة (cron) يومية: تدوير السجل الحالي، نقل القطع القديمة إلى archive/YYYY-MM، ضغط القديم. |
| **طابور سحابي** | إذا CLOUD_SYNC_ENABLED — إضافة السجل إلى طابور رفع؛ عملية خلفية ترفع إلى CLOUD_SYNC_URL أو S3. |

## 4.2 واجهات API (اختيارية للإدارة)

| المسار | الوصف |
|-------|--------|
| **GET /api/operations/status** | إحصائيات: عدد العمليات اليوم، حجم السجل، حالة الطابور السحابي، آخر رفع. |
| **GET /api/operations/essential** | قائمة بالعمليات الضرورية (أو آخر N) — قد يتطلب مصادقة أدمن. |
