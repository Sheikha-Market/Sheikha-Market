# ربط نفاذ وجوجل ومايكروسوفت وأبل لتسجيل الدخول

## التفعيل السريع

1. **تثبيت الحزم:** من مجلد البوابة نفّذ `npm install` (يُضاف حزمة dotenv لتحميل ملف .env).
2. **نسخ ملف البيئة:** انسخ `.env.example` إلى `.env`:  
   `cp .env.example .env`
3. **ملء القيم:** افتح `.env` واملأ على الأقل المزود الذي تريده (مثلاً `GOOGLE_CLIENT_ID` و `GOOGLE_CLIENT_SECRET` من Google Cloud Console).
4. **إعادة تشغيل الخادم:** بعد حفظ `.env` أعد تشغيل الخادم (`npm start` أو `شغل-البوابة.sh`).
5. **التحقق:** افتح صفحة تسجيل الدخول — الأزرار التي تم تعبئة إعداداتها تظهر كمفعّلة؛ أو استدعِ `GET /api/auth/providers` لمعرفة الحالة.

---

## الهدف

تمكين تسجيل الدخول إلى سوق شيخه باستخدام:
- **نفاذ** (Nafath) — الهوية الوطنية الرقمية السعودية
- **جوجل** (Google)
- **مايكروسوفت** (Microsoft)
- **أبل** (Apple — Sign in with Apple)

---

## 1. جوجل (Google Sign-In)

### الحصول على بيانات التطبيق

1. ادخل إلى [Google Cloud Console](https://console.cloud.google.com/)
2. أنشئ مشروعاً أو اختر مشروعاً موجوداً
3. من **APIs & Services** → **Credentials** → **Create Credentials** → **OAuth client ID**
4. نوع التطبيق: **Web application**
5. اسم التطبيق: مثلاً "سوق شيخه"
6. **Authorized redirect URIs** أضف:
   - محلي: `http://localhost:8080/api/auth/google/callback`
   - حي: `https://www.sheikha.top/api/auth/google/callback`
7. انسخ **Client ID** و **Client Secret**

### متغيرات البيئة

```bash
GOOGLE_CLIENT_ID=xxxxx.apps.googleusercontent.com
GOOGLE_CLIENT_SECRET=xxxxx
```

---

## 2. مايكروسوفت (Microsoft / Azure AD)

### الحصول على بيانات التطبيق

1. ادخل إلى [Azure Portal](https://portal.azure.com/) → **Azure Active Directory** → **App registrations** → **New registration**
2. اسم التطبيق: "سوق شيخه"
3. **Redirect URI**: اختر Web وأضف:
   - محلي: `http://localhost:8080/api/auth/microsoft/callback`
   - حي: `https://www.sheikha.top/api/auth/microsoft/callback`
4. بعد الإنشاء: انسخ **Application (client) ID**
5. من **Certificates & secrets** أنشئ **Client secret** وانسخه

### متغيرات البيئة

```bash
MICROSOFT_CLIENT_ID=xxxxx-xxxx-xxxx-xxxx-xxxxxxxx
MICROSOFT_CLIENT_SECRET=xxxxx
```

---

## 3. أبل (Sign in with Apple)

### الحصول على بيانات التطبيق

1. ادخل إلى [Apple Developer](https://developer.apple.com/) → **Certificates, Identifiers & Profiles**
2. **Identifiers** → أنشئ **App ID** (مثلاً com.sheikha.web)
3. **Services IDs** → أنشئ Service ID (مثلاً com.sheikha.web.service) وفعّل **Sign in with Apple**
4. **Keys** → أنشئ مفتاحاً لـ Sign in with Apple وانسخ **Key ID** و **Team ID** و **Service ID**؛ حمّل ملف **.p8** (Private Key)
5. في Service ID أضف **Return URLs**:
   - محلي: `http://localhost:8080/api/auth/apple/callback`
   - حي: `https://www.sheikha.top/api/auth/apple/callback`

أبل يستخدم تدفقاً مختلفاً (نموذج من أبل ثم callback بـ `code` و`id_token`). الخادم يتبادل `code` للحصول على token ويحقّق `id_token` أو يتبادل الـ code مباشرة.

### متغيرات البيئة

```bash
APPLE_CLIENT_ID=com.sheikha.web.service
APPLE_TEAM_ID=xxxxx
APPLE_KEY_ID=xxxxx
APPLE_PRIVATE_KEY_PATH=/path/to/AuthKey_xxxxx.p8
```

## 4. نفاذ (Nafath — الهوية الوطنية الرقمية)

نفاذ هو نظام المصادقة الموحّد في المملكة (الهيئة السعودية للبيانات والذكاء الاصطناعي — SDAIA). التكامل يتم عادة عبر:

- بوابة مزود الخدمة (SP) أو
- استخدام **نفاذ كخدمة** (Nafath as a Service) للمطورين

### الخطوات العامة

1. التسجيل في [منصة نفاذ للمطورين](https://nafath.sdaia.gov.sa/) أو عبر مزود معتمد
2. الحصول على **Client ID** و **Client Secret** (أو ما يوفره النفاذ لبيئة الاختبار/الإنتاج)
3. إضافة **Redirect URI** للموقع (مثلاً `https://www.sheikha.top/api/auth/nafath/callback`)
4. في الخادم: مسار يوجّه المستخدم لصفحة نفاذ، ومسار callback يستقبل الـ token أو الكود ويحقّق الهوية ثم ينشئ أو يربط المستخدم في شيخه

### متغيرات البيئة (عند التفعيل)

```bash
NAFATH_CLIENT_ID=xxxxx
NAFATH_CLIENT_SECRET=xxxxx
```

الوثائق الرسمية والتفاصيل الفنية توجد على بوابة نفاذ للمطورين.

## 5. إعداد الخادم

### قاعدة عنوان الموقع (BASE_URL)

يُستخدم لتشكيل **redirect_uri** في طلبات OAuth. اضبطه حسب البيئة:

- محلي: `http://localhost:8080`
- حي: `https://www.sheikha.top`

```bash
BASE_URL=https://www.sheikha.top
```

أو يكتشف الخادم تلقائياً من طلب المستخدم (Host + Protocol).

### ملف .env (اختياري)

يمكنك إنشاء ملف `.env` في مجلد البوابة (sheikha-main-portal) مع المتغيرات أعلاه. استخدم حزمة مثل `dotenv` لتحميلها عند التشغيل، أو صدّرها في بيئة التشغيل.

## 6. واجهات API المضافة في الخادم

| المسار | الوظيفة |
|--------|----------|
| GET /api/auth/google | يوجّه المستخدم لصفحة تسجيل الدخول بجوجل |
| GET /api/auth/google/callback | يستقبل الكود من جوجل، يتبادل للحصول على token، ينشئ/يربط المستخدم، يوجّه للوحة التحكم مع token |
| GET /api/auth/microsoft | يوجّه المستخدم لصفحة تسجيل الدخول بمايكروسوفت |
| GET /api/auth/microsoft/callback | نفس الفكرة لمايكروسوفت |
| GET /api/auth/apple | يوجّه لتسجيل الدخول بأبل (أو يرد معلومات للمستخدم لاستخدام زر أبل من الواجهة) |
| GET /api/auth/apple/callback | يستقبل callback من أبل |
| GET /api/auth/nafath | يوجّه لتسجيل الدخول بنفاذ (أو يرد رابط/معلومات للتكامل) |
| GET /api/auth/nafath/callback | يستقبل callback من نفاذ |

بعد نجاح أي مسار callback، الخادم يوجّه المتصفح إلى صفحة الدخول مع `?token=JWT` أو يرسل الـ token بطريقة آمنة أخرى؛ الواجهة تحفظ الـ token (مثلاً في localStorage) وتوجّه إلى لوحة التحكم.

---

## 7. صفحة تسجيل الدخول

في **تسجيل-الدخول.html** تمت إضافة قسم «أو الدخول باستخدام» مع أزرار:

- الدخول بنفاذ
- الدخول بجوجل
- الدخول بمايكروسوفت
- الدخول بأبل

كل زر يوجّه إلى المسار المناسب أعلاه (نفس النطاق). بعد العودة من مزود الهوية، الصفحة تقرأ الـ token من الـ URL وتوجّه إلى لوحة التحكم.

---

## 8. المستخدمون من OAuth

عند أول دخول عبر جوجل/مايكروسوفت/أبل/نفاذ:

- يُنشأ **مستخدم جديد** في قاعدة المستخدمين مع حقول: `provider`، `providerId`، والاسم والبريد من مزود الهوية
- يُنشأ له **تاجر** مرتبط (إن وُجد منطق إنشاء تاجر تلقائي)
- يُصدر **JWT** ويُوجّه إلى لوحة التحكم

الدخول اللاحق بنفس الحساب (نفس provider + providerId) يربط الجلسة بنفس المستخدم دون إنشاء مكرر.
