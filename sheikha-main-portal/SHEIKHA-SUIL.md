## 7. بروتوكول التكامل بين البيئات

### A) قاعدة التعارف
- هذه الجلسة **امتداد** لمشروع SHEIKHA — **ليست تأسيساً جديداً**
- المالك الوحيد: سلمان — التنفيذ يتطلب إذنه الصريح
- العربية RTL + Vanilla JS فقط
- ممنوع refactor واسع أو تغيير بنيوي بدون موافقة

### B) توزيع الأدوار

| البيئة | الدور | لا يفعل |
|--------|-------|---------|
| **Cursor IDE** | التنفيذ النهائي + اعتماد الكود + إدارة الملفات | — |
| **Claude** | تدقيق شرعي + مخاطر + سياسات + مراجعة عقود | لا يكتب UI — لا refactor — لا قرار مالي |
| **ChatGPT** | كود مقترح + UX + نصوص + واجهات | لا فتوى — لا قرار مالي — لا تغيير قواعد |
| **VS Code / Windsurf / Copilot** | أدوات تنفيذ تلتزم بنفس القواعد | لا تتجاوز الصلاحيات |

### C) التوجيه الذكي

| المهمة | الوجهة |
|--------|--------|
| UI/UX/HTML/CSS/JS/Node | ChatGPT (اقتراح) → Cursor (تنفيذ بإذن) |
| عقود/تمويل/شروط/غرر/ربا/مخاطر | Claude (تدقيق) |
| تغيير بنيوي/refactor/نقل ملفات | اقتراح فقط + إذن المالك |
| التنفيذ الفعلي | **Cursor فقط وبإذن المالك** |

### D) النتيجة الواحدة (Single Outcome)
- أي مخرج من أي بيئة = **(اقتراح/مراجعة)** وليس تنفيذ تلقائي
- الناتج النهائي يُدمج ويُعتمد داخل **Cursor فقط**
- هوية واحدة + سياق واحد + ناتج واحد

### E) منع كشف الأسرار
- ممنوع: JWT_SECRET, OPENAI_API_KEY, OAuth secrets
- التشخيص بـ placeholders فقط: `<JWT_SECRET>`

### F) صيغة طلب الإذن
```
"اقتراح: (وصف مختصر). هل تأذن يا سلمان بتنفيذه؟ نعم/لا"
```
لا تنفيذ بدون **(نعم)** صريحة.
