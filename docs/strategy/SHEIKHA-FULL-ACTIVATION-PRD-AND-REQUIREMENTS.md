# SHEIKHA — PRD & Architecture Requirements for Full Activation

## الاعتماد الرسمي

هذه الوثيقة تعتمد متطلبات التفعيل الكامل لمنظومة شيخة ضمن نطاق قابل للقياس والتنفيذ المرحلي، وتعد المرجع الرسمي للمتطلبات الوظيفية وغير الوظيفية.

## 1) تعريف "التفعيل الكامل" (Scope Definition)

التفعيل الكامل يعني تحقيق جاهزية تشغيلية وأمنية وذكائية ومعمارية وتكاملية ضمن حدود المنصة، وفق حوكمة واضحة ومؤشرات قياس معتمدة.

### 1.1 الأبعاد الإلزامية للتفعيل

1. الجاهزية التشغيلية (Operational Readiness)
2. الجاهزية الأمنية والامتثال (Security & Compliance Readiness)
3. جاهزية محرك الذكاء والشبكة العصبية (AI/Neural Readiness)
4. الجاهزية المعمارية (Architecture Readiness)
5. الجاهزية التكاملية المحلية والخارجية (Integration Readiness)

### 1.2 حدود النطاق

- داخل النطاق:
  - خدمات السوق الأساسية (Catalog, Suppliers, Orders, Analytics, Auth)
  - دورة حياة تشغيل الخلايا العصبية ضمن حدود المنصة
  - المراقبة والتنبيه واستمرارية الخدمة
  - CI/CD وحوكمة الإصدارات
- خارج النطاق (في هذه المرحلة):
  - أي ادعاءات تشغيلية خارج حدود الأنظمة المتصلة فعليًا
  - أي تكاملات غير موثقة بعقود API وتشغيل

## 2) المتطلبات الوظيفية (Functional Requirements)

### 2.1 متطلبات المنصة الأساسية

- FR-API-01: توفر نقاط الصحة والحالة (`/api/health`, `/api/v2/health`, `/api/v2/status`) بحالة قابلة للتحقق.
- FR-API-02: إدارة المنتجات والموردين والتحليلات عبر واجهات موثقة.
- FR-AUTH-01: تسجيل ودخول JWT مع سياسات صلاحيات واضحة حسب الدور.
- FR-ORD-01: حماية نقاط الطلبات الخاصة ومنع الوصول غير المصرح.
- FR-OBS-01: تسجيل أحداث تشغيلية أساسية بدون كشف بيانات حساسة.

### 2.2 متطلبات محرك الذكاء والشبكة العصبية

- FR-AI-01: توفير حالة جاهزية الخلايا الأساسية وإجمالي الخلايا من نقطة موحدة.
- FR-AI-02: تعريف دورة حياة واضحة للخلايا: Initialize → Validate → Activate → Monitor → Recover.
- FR-AI-03: إلزام عقود إدخال/إخراج ثابتة لواجهات الاستدلال.
- FR-AI-04: توفير مؤشرات إنذار عند انخفاض الجاهزية أو فشل التفعيل.

### 2.3 متطلبات التكامل

- FR-INT-01: كل تكامل خارجي يجب أن يمر بعقد واجهة API + SLA + سياسة مفاتيح.
- FR-INT-02: التحقق من صحة البيانات الواردة والصادرة لكل موصل.
- FR-INT-03: تسجيل حالات الفشل وإعادة المحاولة وفق حدود آمنة.

## 3) المتطلبات غير الوظيفية (Non-Functional Requirements)

### 3.1 الأمن والامتثال

- NFR-SEC-01: تفعيل حماية HTTP headers وسياسات الحماية الأساسية.
- NFR-SEC-02: تطبيق Rate Limiting على نقاط النهاية العامة والحساسة.
- NFR-SEC-03: عدم تسجيل كلمات المرور أو الرموز السرية أو JWT في السجلات.
- NFR-SEC-04: تطبيق فحوص أمن تبعيات دورية ضمن CI.

### 3.2 الأداء والموثوقية

- NFR-REL-01: توافر تشغيلي مستهدف لا يقل عن 99.5% في المرحلة الأولى.
- NFR-PERF-01: P95 لواجهات الصحة والحالة ضمن حدود تشغيلية معتمدة.
- NFR-REL-02: خطط استعادة واضحة للأخطاء الحرجة (Runbook + Escalation).

### 3.3 القابلية للرصد والحوكمة

- NFR-OBS-01: مؤشرات موحدة للصحة، الأخطاء، والجاهزية العصبية.
- NFR-GOV-01: كل تغيير جوهري يمر عبر مراجعة معمارية وأمنية.
- NFR-GOV-02: سجل قرارات معتمد (ADR/Decision Log) للتغييرات المؤثرة.

## 4) معايير القبول الشاملة (Global Acceptance)

- AC-01: نجاح فحوص CI الأساسية بدون كسر خطوط الجودة.
- AC-02: تحقق نقاط الصحة والحالة والاستقرار ضمن سيناريوهات الاختبار.
- AC-03: تحقق سلامة دورة الحياة للخلايا العصبية وفق عقود واضحة.
- AC-04: توفر لوحات متابعة KPI وتقارير دورية قابلة للمراجعة.
- AC-05: وجود خطة تشغيل ومخاطر معتمدة وقابلة للتنفيذ.

## 5) الحوكمة الشرعية والعملية

- منع الربا والغرر والمنتجات المحرمة ضمن قواعد المنصة.
- منع الوعود المضللة أو الادعاءات غير القابلة للتحقق.
- إبقاء القرار البشري إلزاميًا في القرارات الحساسة.
